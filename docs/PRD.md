# PRD.md

## 项目简介

Unveiling 是一个多 Agent 类比分析系统。用户带着问题或情境来（而非孤立主题），系统通过横向（跨领域）和纵向（跨时期）类比，以及抽象化的透镜思维，揭示事物深层结构性模式，产出张力式洞察。

## 核心哲学

### 根基

**1. 结构比表面更真实（解释力来自结构层，不来自现象层）**

"深层结构上的映射"比"表面相似"更有价值。如: AI 和印刷术都"引发了公众恐慌"是现象层共性；"都使某类认知劳动的边际成本趋近于零"是结构层共性——后者有迁移性和预测力，前者没有。

> 为什么结构层有迁移性——因为结构层描述的是关系和机制，而关系和机制可以在不同的表面材质上实例化。

**2. 事物的普遍性**

万物在深层结构上存在跨时空的映射。面对任何问题，时空的另一角落往往存在结构相似且值得寻找的案例。通过研究这些远距离案例，识别其深层结构，映射回当前问题，可以洞见当前问题的本质。但如果当找到的类比都停留在现象层、无法提供结构性理解时，这本身就是一个输出——说明这个问题可能是真正的新事物。

**3. 距离产生清晰（"不识庐山真面目，只缘身在此山中"）**

为什么不直接分析当前问题？因为身处其中时，偏见和盲区阻碍了真正的理解。借助时空中的远距离案例进行类比，我们获得外部视角，才能看到身处其中时看不到的结构。

"类比是认识的**入口**，直接分析是类比之后的**落地**。系统优先通过类比建立结构性理解，在此基础上才触碰当前问题的具体细节——而不是反过来。"

**4. 远近张力原则**

认知价值和认知可达性之间存在内在张力：最有价值的类比往往最难被识别为相关。透镜是这个张力的操作解：它不消除距离，而是在保持距离的同时建立结构映射，让遥远的案例在抽象层上变得可比。

透镜本身随分析过程演化——初始透镜是进入的入口，演化后的透镜是分析的结论之一。其演化由两种力量驱动：近搜索带来的结构确认，和远搜索带来的结构挑战。当一个远距离案例无法被当前透镜解释但直觉上重要时，这是透镜需要演化的信号。

基于该原则，系统应该显式地同时维护"远"和"近"两种搜索模式，而不是只有一种。

- "近"搜索：在当前透镜框架内寻找结构匹配的类比——高命中率，低惊喜
- "远"搜索：主动寻找当前透镜框架解释不了、但直觉上感觉有趣的案例——低命中率，高潜在价值

两种搜索的配比，就是"问题导向"和"自由联想"张力的实操版 trade-off。可以把它变成一个可调的旋钮，就像是模型的 temperature 参数一样。

### 衍生原则

基于上述根基：

**1. 工具论：用什么思考**

时空类比和透镜是系统的核心认知工具，不绑定到特定阶段，随时可调用。

两者的关系是：类比提供原材料，透镜提供结构映射的框架。没有透镜，遥远的类比无法被评估；没有类比，透镜是空转的抽象。

透镜不是分析的前提，而是分析的产物。初始透镜是进入问题的脚手架，随着类比证据的积累，透镜被修正、分裂或合并——其演化轨迹本身记录了理解的发生过程，是洞察的一部分。

**2. 证据论：如何对待发现**

所有发现的共性都不丢弃，但追问方式因性质而异。

系统不预先判断一条共性的真假，而是追问三个问题：它在哪个层级上成立（phenomenon、mechanism、还是 structure）？在当前问题框架下是否有解释力？它为什么看起来成立——或为什么看起来不成立？

这个追问过程本身产生洞察，无论最终结论如何：真共性揭示结构，假共性揭示认知盲区，层级判断本身校准透镜。

**3. 导向论：以什么为罗盘**

以问题为锚，以结构性理解为目标，以意外发现为信号。

"以问题为锚"不意味着只追求直接相关的类比。"看起来无关"本身是一个需要透镜才能做出的判断，而透镜还在演化中——过早放弃远距离类比，可能是用一个尚未成熟的透镜关上了一扇门。

因此：近搜索（透镜框架内的结构匹配）保证问题被回答，远搜索（框架外的意外发现）保证透镜被挑战。两者的配比是可调的，但两者都不可缺席。

## 架构方向

### 设计原点

核心哲学确立了两件事：分析应以问题为锚，以结构性理解为目标。这直接推出架构的基本形状：需要一个认知层来维护"我们在回答什么"，需要一个执行层来调用"我们用什么工具回答"。两层分离，职责不混。

------

### 两层架构

两层架构外有 Orchestrator 作为控制层，负责两层之间的协调，详见后文。

**认知层：问题驱动**

认知层是系统的罗盘，负责维护分析的方向感。它的核心数据结构是问题树（issue tree）：

```
driving_question（根）
├── sub_question_1
├── sub_question_2
└── sub_question_3
```

问题树决定"分析什么"，是 Orchestrator 调度的唯一界面——每一轮调度的起点都是"哪个子问题还没有被充分回答"，而不是"还有什么可以挖"。

认知层同时维护可证伪预判。每个预判的格式是：

- claim（主张）
- if_true_we_should_see（如果成立，应看到）
- if_false_we_should_see（如果不成立，应看到）
- killer_evidence（什么单条证据能直接推翻它）

预判的功能是 hypothesis-driven research 的脚手架：它告诉执行层"去找什么"，也告诉收敛阶段"用什么标准评估发现"。

预判的生命周期状态：pending → supported / refuted / modified。状态由收敛期的预判回顾步骤判定。

**执行层：能力调度**

执行层是系统的手，负责实际的探索和发现。它提供三类平级能力，由 Orchestrator 按需调用：

- 透镜：建立结构映射框架，随时可生成 / 修正 / 分裂 / 合并，演化轨迹完整保留
- 纵向发现：跨时期的历史先例搜索
- 横向发现：跨领域的结构类比搜索

纵向和横向搜索都显式包含两个任务：在透镜框架内寻找结构匹配（近搜索），以及主动寻找框架解释不了但直觉上重要的案例（远搜索）。所有搜索任务都显式包含寻找 killer_evidence，不只是寻找确认性证据。

------

### 远近搜索配比

远近张力原则在架构层的落地是一个可调参数，控制近搜索（框架内结构匹配）和远搜索（框架外意外发现）的资源分配比例。

用户侧：用户可以在启动时设定初始偏好，也可以在分析过程中随时调整。

- 偏近（"我要回答一个具体问题"）→ 系统优先在透镜框架内寻找结构匹配，快速收敛到可回答问题的答案
- 偏远（"我想看看有什么意想不到的角度"）→ 系统主动搜索框架外案例，容忍更多不确定性，换取更高的惊喜概率
- 不设定 → 交给 Orchestrator 自动管理

用户不需要理解"远近搜索"的技术含义，界面上可以表达为一个直觉性的选择："聚焦回答" ←→ "开放探索"。

Orchestrator 侧：在用户设定的偏好范围内，Orchestrator 根据问题树的当前状态动态微调：

- 分析早期，透镜尚未成熟 → 偏远：拓宽视野，让透镜接受挑战
- 分析中期，已有发现但子问题未闭合 → 平衡：近搜索填补证据，远搜索防止过早收敛
- 分析后期，接近收敛 → 偏近：确认结构、补齐证据链
- 特殊情况：子问题长期 stuck，或意外发现密集积累 → 临时调高远搜索，可能触发透镜演化或问题树重构

配比不是全局统一的——不同子问题可以有不同的远近比例。一个已有强结构性理解的子问题不需要再远搜索；一个完全 stuck 的子问题可能恰恰需要跳出当前透镜。

------

### 深挖机制

深挖是执行层的核心循环，但它受认知层约束：深挖的对象由问题树的当前优先子问题决定，不是"挖到什么算什么"。

深挖的每一步带层级标签：phenomenon → mechanism → structure

规则：

- 同一层不超过 2 步，必须主动尝试跨层
- 目标不是挖得更深，而是挖到有迁移性的 structure 层

跨层失败时的处理：

- 明确标记"此线索止于 X 层"，记录尝试过什么跨层追问、为什么推不动
- 这不是失败——一条共性止于 mechanism 层，本身是有价值的发现，说明它的解释力有边界
- "推不动"的原因写入辩论区，可能成为其他子问题的线索

对所有发现的共性，不预先判断真假，而是追问：它在哪个层级成立？在问题框架下是否有解释力？它为什么看起来成立或不成立？——这个追问过程本身产生洞察。

------

### 全局黑板

黑板是两层架构的共享状态，分六个分区：

- 问题区：问题树。系统的骨架，Orchestrator 的调度界面。每个节点记录当前回答状态（untouched / exploring / closed / stuck）。
- 假设区：透镜版本链 + 可证伪预判。透镜记录演化轨迹（生成 / 修正 / 分裂 / 合并的触发原因和基于的证据）。预判记录 claim、if_true/if_false_we_should_see、killer_evidence、status（pending / supported / refuted / modified）。
- 证据区：搜索结果、具体案例。每条带元数据：来源透镜及版本、所属子问题、层级标签（phenomenon / mechanism / structure）、置信度（strong / medium / weak / unexpected）。只增不删。
- 辩论区：质疑、反驳、深挖追问记录、跨层失败的原因。只增不删。
- 结论区：收敛阶段写入的张力式输出。
- 调度日志区：Orchestrator 的关键决策痕迹（回修决策、配比切换、子问题状态变更、LLM 失败降级）。只增不删。

各分区的角色定位：

- 问题区是认知层的骨架
- 假设区是认知层的工具和命题
- 证据区和辩论区是执行层的工作台
- 结论区是两层协作的最终产物
- 调度日志区是 Orchestrator 的决策记录，事后复盘和向用户呈现可靠性边界时使用

Orchestrator 的调度循环以问题区为起点，读取假设区做决策，读取证据区和辩论区判断饱和度和覆盖度。

#### 黑板的轮询式协作机制

黑板不是实时并发的共享空间，而是通过"短任务 + 高频调度循环"实现近实时的协作效果。核心思路：把所有 Agent 的任务切成小颗粒（每次任务几秒到十几秒），任务完成后提交 State，Orchestrator 在下一轮调度时立即看到变化并做出响应。

实际效果不是真正的实时，但延迟在 LLM 调用粒度内——本来每次调用就要几秒到十几秒，在这个尺度下，调度循环的延迟几乎无感。

**任务粒度原则**

- 每个执行节点的任务粒度小：一次搜索、一次深挖追问、一次透镜修正，各自是一个独立任务
- 独立的子问题推进可以用 LangGraph 的并行分支同时执行
- 任务完成即提交：节点执行结束时，结果写入 State，对 Orchestrator 立即可见

**调度循环**

Orchestrator 不"持续监听"，而是在每个任务完成后运行一次决策循环：

1. 读取 State 的当前状态（问题树、假设区、证据区、辩论区的最新内容）
2. 运行决策（Scheduler → 必要时 Judge → 必要时 Meta）
3. 派发下一轮任务

"链式反应"通过 Scheduler 在每轮重新评估实现：任务 A 完成提交新证据 → Scheduler 看到新证据 → 决定派发对此证据的反应任务。延迟在 LLM 调用粒度内，对分析过程来说感知上接近实时。

**记录状态**

每条黑板记录的状态简化为两级：

- committed：记录已稳定，可被其他 Agent 安全引用。所有进入黑板 State 的记录默认是 committed——因为它们是任务完成后才提交的，不存在"中途写入"。
- retracted：已撤回。记录本身不删除（保留追溯性），但标记为"不再有效"，并记录撤回原因。

draft 状态不再作为跨 Agent 可见的状态存在。Agent 内部的推敲过程不暴露到黑板——任务粒度足够小，Agent 完成推敲后才提交结果。这简化了状态机，也消除了"基于不稳定内容工作"的风险。

**撤回的处理**

当一条记录被标记为 retracted 时：

- 系统自动识别哪些后续记录引用了它
- 这些后续记录被标记为"前提已撤回，需重审"
- Orchestrator 收到通知，决定是否启动重审任务，还是接受后续记录的失效

撤回不是失败——一个 Agent 主动撤回自己的输出，比让错的内容污染整个分析更负责任。撤回的痕迹和原因写入辩论区。

**追溯性**

每条记录带完整元数据：写入者、时间戳、当前状态、状态变更历史、触发原因、引用了哪些已有记录。事后可以重建任何一个时刻的黑板状态。LangGraph 的 Checkpoint 机制天然支持这一点。

Orchestrator 的角色是"在每个任务完成后评估全局状态，决定下一步做什么"——高频轮询式的决策循环，而不是持续监听。

### Orchestrator 自身的设计

#### 定位

Orchestrator 不是一个"做所有决策的全能 Agent"，而是一个有限状态机 + LLM 判断点的混合控制器。

- 状态机部分：保证流程的可预测性、可调试性、token 消耗可控
- LLM 判断点：处理真正需要判断力的事情——那些规则写不清、必须依赖语义理解的决策

这个分工的核心信念是：让规则做规则擅长的事，让 LLM 做 LLM 擅长的事。如果一个决策可以用规则清晰表达，就不要浪费 LLM 调用；如果一个决策本质上需要语义判断，就不要用脆弱的规则去硬套。

#### 内部分工：三个角色

Orchestrator 内部分为三个角色，各自承担不同性质的决策。三者共享黑板状态，但 LLM 调用相互独立——这样每个角色的 context 聚焦，决策互不污染。

**Scheduler（调度官）**

负责"下一步做什么"。

- 输入：问题树当前状态、各子问题的最近活动
- 输出：下一个能力调用任务（调用什么、给哪个子问题、远近偏好）
- 决策风格：规则为主，LLM 为辅
- LLM 调用场景：当多个子问题都需要推进、且优先级难以用规则区分时，让 LLM 判断"现在最该推进哪个"

**Judge（裁判）**

负责"现在到了什么状态"。

- 输入：某个子问题的全部相关证据、辩论、深挖记录
- 输出：该子问题的状态判断（exploring / closed / stuck）、minimum viable answer 是否达成
- 决策风格：LLM 为主，规则为辅
- 规则兜底：硬性条件（如"完全没有 structure 层发现"）直接判定未达成，不用浪费 LLM 调用

**Meta（元认知者）**

负责"是否需要回修"。

- 输入：黑板的全局趋势（最近 N 轮的发现密度、意外发现的累积、透镜挑战次数等）
- 输出：是否触发回修、触发哪种回修（问题树重构、新透镜、stuck 子问题再尝试）
- 决策风格：纯 LLM
- 调用频率：低（每 N 轮或达到特定信号阈值时调用一次），因为这是最贵也最重要的决策

三个角色的调用频率从高到低：Scheduler（每个任务完成都调用） > Judge（子问题状态可能变化时调用） > Meta（趋势性变化才调用）。

**上下文管理说明**：当子问题相关内容过多时，Judge 和 Meta 需要先做一次摘要再判断——这是一个工程实现细节，但摘要本身可能丢失关键信息，需要在 PoC 阶段验证。

#### 触发机制

Orchestrator 的主循环由两种事件驱动：

1. 任务完成触发：每个执行 Agent 完成任务时，向 Orchestrator 发出完成信号。Orchestrator 同步运行一次决策循环（Scheduler → 必要时 Judge → 必要时 Meta）。
2. 定时兜底：定时器每 N 秒检查一次系统状态，防止系统僵死（比如所有 Agent 都在等待但没人推进）。

这个机制在 LangGraph 上的实现是天然的：

- 任务完成 = 节点执行结束 → 自动流转到 Orchestrator 节点
- 定时兜底 = 一个并行的监控节点，定期检查状态并按需注入决策

#### 决策类型：规则与 LLM 的分界

明确划分哪些决策走规则、哪些走 LLM，是 Orchestrator 设计的核心。

走规则的决策（确定性强、可枚举）：

- 是否进入收敛期（所有叶子节点 closed 或 stuck）
- 是否触发硬性兜底（token 预算、最大轮次）
- 远近配比的阶段默认值（早期偏远、晚期偏近）
- minimum viable answer 的硬性条件检查（是否有 structure 层发现、是否经历过质疑回合）
- 撤回引发的级联失效识别（依赖图遍历）

走 LLM 的决策（语义判断、不可枚举）：

- 多个子问题优先级难分时，选哪个推进
- 子问题的实质性状态判断（minimum viable answer 是否达成，规则只能查硬性条件，LLM 判断是否真的"答到了"）
- 意外发现是否暗示问题树拆错了
- 一个看似无关的远搜索结果是否值得深挖
- 收敛期某个 tension 是否值得回到探索期补刀（边际收益评估）

混合决策（规则筛 + LLM 判）：

- 远近配比的实时微调：规则给出阶段基准值，LLM 在用户偏好范围内基于黑板状态微调
- 子问题 stuck 判定：规则检查"尝试次数 ≥ N 次"，LLM 判断"是否真的没有新角度可试"

#### 状态管理：内部 vs 黑板

Orchestrator 的状态分两类：

写入黑板的（关键决策，需要追溯）：

- 每次回修决策（"在 t 时刻决定问题树重构，触发原因是 X"）
- 每次远近配比的阶段切换（"在 t 时刻从平衡切换到偏近"）
- 每次子问题状态变更（"sub_question_2 在 t 时刻被判定为 stuck，原因是 Y"）
- LLM 判断失败导致的降级处理（"在 t 时刻 Meta 调用失败，采用保守默认值继续探索期"）

这些写入"调度日志区"（见"全局黑板"小节）——其价值在于事后复盘和调试。

保留在内部的（临时状态，不污染黑板）：

- 每个子问题的当前尝试次数计数器
- 最近 N 轮的发现密度滑动窗口
- 当前 Scheduler 等待的执行中任务列表
- 定时兜底的下次触发时间

这些是 LangGraph 的 State 对象的一部分，跟着图执行流转，但不写入持久化黑板。

#### 失败降级

LLM 调用会失败：超时、返回格式错误、返回明显不合理的判断。Orchestrator 作为系统大脑，必须有降级策略。

通用降级原则：LLM 失败时，降级为保守默认值，继续运行而不是终止，并在黑板调度日志区明确标记降级事件。

具体策略：

| 失败场景               | 降级行为                                                     |
| :--------------------- | :----------------------------------------------------------- |
| Scheduler 选不出下一步 | 按问题树顺序选择第一个未闭合的子问题；连续 N 次降级时触发 Meta 评估是否需要回修 |
| Judge 判断不出状态     | 保持当前状态不变，下一轮证据后重判；<br />连续 N 次失败 → 强制标记为 stuck（原因：状态判定失败） |
| Meta 判断不出是否回修  | 默认不回修，继续当前流程                                     |
| LLM 重试 N 次仍失败    | 该决策点彻底降级为规则兜底，并标记"此轮分析的某些决策因 LLM 不可用而退化" |

降级标记不是装饰——收敛期的最终输出应该向用户呈现"本次分析中有 X 处决策走了降级路径"，让用户知道结果的可靠性边界。

#### 与 LangGraph 的对接

LangGraph 的 StateGraph 模型天然承载这套设计：

- 节点（Node）：执行 Agent（搜索、深挖、透镜操作、辩论、预判检验各是一个节点），Orchestrator 的三个角色（Scheduler / Judge / Meta）也各是一个节点
- 状态（State）：黑板 + Orchestrator 内部状态。LangGraph 的 State 对象天然支持共享和增量更新
- 边（Edge）：执行 Agent 完成后流转到 Scheduler；Scheduler 根据决策路由到下一个执行 Agent；Judge 和 Meta 由 Scheduler 按需调用
- 条件路由（Conditional Edge）：状态机的核心——基于 State 的字段决定下一步走哪条边
- 检查点（Checkpoint）：LangGraph 内置的状态持久化机制天然支持黑板的追溯性
- 中断与恢复（Interrupt）：用户介入点（启动期确认、过程中调整偏好）可以用 LangGraph 的 interrupt 机制实现

这个对接的核心好处是：Orchestrator 的状态机不需要自己实现，LangGraph 的图执行引擎就是状态机。我们只需要定义节点、状态、条件路由，LangGraph 处理调度和持久化。

### 收敛与输出

收敛条件由问题树驱动，不是"每个透镜 N 条证据"这种魔法数字。

一个子问题的 minimum viable answer 需要满足：

- 至少有一条 structure 或 mechanism 层的发现（不能全停留在 phenomenon 层）
- 相关的可证伪预判已被检验（killer_evidence 找到了 / 没找到，都算检验）
- 在辩论区至少经历过一次质疑回合

所有叶子节点达到 minimum viable answer，系统进入收敛。

硬性兜底：达到全局 token 预算或最大调用轮次，强制收敛，未闭合的子问题在输出中明确标记为"未完成"。

输出结构强制为张力式，不是共性堆叠。

对每个子问题：

- convergent_finding：多角度支持的结论
- tension：不同角度的冲突点（洞察高浓度区）
- boundary_condition：结论在什么条件下成立、在什么条件下崩坏
- unresolved：本子问题中还未回答清楚的部分，以及"推不动"的地方
- implication：对 driving_question 的回答贡献了什么

tension 和 unresolved 都是强制项。

- 找不到 tension：要么分析不够深，要么问题本身没有张力——两种情况都应该显式输出，不用共性堆叠掩盖
- 没有 unresolved：要么问题太浅，要么系统过度自信——分析员的诚实在于知道自己不知道什么

最终输出回到 driving_question：把各子问题的 implication 综合起来，回答最初的驱动性问题，并明确指出这个回答的边界和未尽之处。

## 分析流程

### 生命周期概览

一次完整分析分三期：启动期、探索期、收敛期。

三期不是固定步骤，而是状态——由黑板和问题树的当前状态决定系统处于哪一期。三期之间不是单向流动：探索期可以触发对启动期产物的回修（问题树重构、透镜大幅演化），收敛期发现关键缺口时也可以临时回到探索。

默认流向：启动期 → 探索期 → 收敛期。

回修通道：
- 探索期 → 启动期（问题树重构、生成新透镜）
- 收敛期 → 探索期（补刀关键证据）
- 收敛期不直接回到启动期

每一期的描述包括：目标、入口条件、典型行为、出口条件、远近搜索特征。

------

### 启动期

目标：建立认知层的初始骨架，让后续探索有锚、有方向、有可证伪的标靶。

入口条件：用户输入到达。输入包含三部分：

- 问题或情境（必须）
- 决策语境或使用场景（可选但强烈建议）
- 远近偏好设定（可选，未设则交给 Orchestrator）

典型行为：

1. 澄清 driving_question。如果用户输入已经是清晰的问题，直接采纳；如果是模糊情境（"我想了解 AI 像不像印刷术"），系统生成 2-3 个候选 driving_question 让用户选择。这一步本身就是有价值的——它逼用户搞清楚自己想要什么。
2. 拆解问题树。把 driving_question 拆为若干子问题，要求 MECE 倾向（互斥、穷尽不强求，但要避免明显遗漏）。每个子问题写入问题区，初始状态标记为 untouched。
3. 生成初始透镜。基于 driving_question 和子问题，生成 2-3 个初始透镜。明确标注这些透镜是脚手架性质——它们的作用是开启探索，不是给出答案，预期会在探索期被大幅修正。
4. 生成可证伪预判。每个预判按规定格式：claim、if_true_we_should_see、if_false_we_should_see、killer_evidence。如果一个预判写不出 killer_evidence，说明它太模糊或不可证伪，需要重写或舍弃。
5. （交互模式）用户确认。展示问题树、初始透镜、预判，让用户修正或追加。非交互模式直接通过。

出口条件：

- 问题树就位，至少有一层子问题
- 至少有一个初始透镜
- 至少有一条可证伪预判（带 killer_evidence）
- 用户确认通过（交互模式）

远近搜索特征：偏远。因为透镜还不成熟，偏近会过早把分析锁死在初始框架内。Orchestrator 在启动期默认拉高远搜索权重，除非用户明确设定为"聚焦回答"。

------

### 探索期

目标：让问题树的每个叶子节点接近 minimum viable answer。

入口条件：启动期出口达成。

典型行为（无固定顺序，由 Orchestrator 基于黑板状态动态调度）：

Orchestrator 在每个任务完成后重新评估：当前问题树状态下，哪个子问题最需要推进？是否需要启动新的能力调用？是否需要停止某个进行中的任务（因为它的前提已经被新证据推翻）？可调用的能力组合包括：

- 搜索类：纵向发现（跨时期）、横向发现（跨领域），每次显式包含近搜索任务、远搜索任务、killer_evidence 寻找任务
- 深挖类：对已发现共性逐层追问，带层级标签（phenomenon / mechanism / structure），同层不超 2 步，必须尝试跨层
- 透镜操作：生成新透镜、修正现有透镜、分裂、合并。每次操作记录触发原因和基于的证据
- 辩论类：调度多个 Agent 对某条共性或某个预判进行质疑—回应回合
- 预判检验：基于已积累的证据，检查可证伪预判是否被支持、推翻或部分修改

每一轮调度后，Orchestrator 更新问题树的节点状态：

- untouched → 完全没探索过
- exploring → 已有部分发现但未达 minimum viable
- closed → 达到 minimum viable answer
- stuck → 尝试多次但无法推进，标记原因

回修触发：回修不是被某一轮"决定"触发的，而是黑板状态自然演化到某个临界点时触发的。可能的触发条件示例：

- 黑板上意外发现密集积累，暗示当前问题树拆错了 → 问题树重构
- 多个子问题都需要某个新概念才能推进 → 生成新透镜
- 某个初始透镜被证据反复挑战 → 透镜大幅演化（可能升级为新透镜）

回修不是失败，而是分析在自我校准。回修的痕迹写入辩论区，本身是本次分析输出的一部分。

出口条件：

- 所有叶子节点状态为 closed 或 stuck，或
- 触发硬性兜底（token 预算、最大调用轮次）

stuck 也算出口条件——一个子问题确实回答不了，比让系统无限循环更诚实。stuck 子问题在收敛期会显式标记。

远近搜索特征：动态调节。

- 早期偏远（透镜未成熟）
- 中期平衡（既要填证据又要防过早收敛）
- 晚期偏近（确认结构、补齐证据链）
- 子问题级别的差异：已有强结构性理解的子问题偏近，stuck 子问题临时拉高远搜索

所有调节都在用户设定的偏好范围内。

------

### 收敛期

目标：把探索期积累的发现收拢为张力式输出，回答 driving_question。

入口条件：探索期出口达成。

典型行为：

1. 溯源验证。对每条进入结论的发现，追溯到证据区的具体来源。无法溯源的发现降级或舍弃。
2. 预判回顾。对启动期生成的每条可证伪预判，检查：
   - killer_evidence 找到了吗？
   - if_true / if_false 的证据形态出现了哪些？
   - 预判状态：supported、refuted、还是 modified？
   - 哪些预判在探索期被新预判替代？
3. 透镜演化总结。展示透镜从初始版本到最终版本的演化轨迹。关键问题：随着透镜演化，证据区呈现的画面发生了什么变化？这本身往往是高浓度洞察。
4. 跨子问题的张力识别。这是收敛期最关键的一步。把各子问题的发现并排，主动寻找：
   - 不同子问题的结论之间是否冲突？
   - 同一案例在不同子问题下是否呈现不同面貌？
   - 如果存在冲突，是真实的张力（值得保留为洞察）还是分析瑕疵（需要修正）？
5. 生成张力式输出。对每个子问题填写五项：convergent_finding、tension、boundary_condition、unresolved、implication。tension 和 unresolved 是强制项。
6. 回答 driving_question。综合所有子问题的 implication，回答最初的驱动性问题。明确指出：
   - 这个回答的边界条件
   - 哪些子问题未闭合（stuck）
   - 哪些方向值得后续探索

回修触发：黑板状态达到"某个 tension 重要但证据不足"时自动触发，临时返回上一阶段。可能的触发条件示例：

- 某个 tension 看起来重要但证据不足以判断真伪 → 针对性的远搜索补充
- 某条 killer_evidence 从未被认真寻找过 → 补一次定向搜索
- stuck 子问题在收敛期如果出现新线索（如其他子问题的发现意外提供了角度） → 可以选择性地再尝试一次，但需要 Orchestrator 判断边际收益。

回修是有限的，受 token 预算约束。Orchestrator 需要判断"补这一刀的边际收益是否值得"。

出口条件：

- 结论区完成，覆盖所有子问题
- driving_question 得到回答（即使回答是"现有信息无法回答"）
- 输出的边界和未尽之处明确标记

远近搜索特征：偏近，但允许为补齐关键证据做针对性的远搜索。这不是矛盾——大方向上系统在收拢，但收拢过程中发现关键缺口时，定向的一次远搜索比放任系统在已有证据里反复打转更有价值。

------

### 三期之间的协作关系

三期不是流水线，而是有主导期但允许回流的状态机。任何时候系统处于某一期，意味着这一期的目标是当前主导目标，但其他期的行为不被禁止——只是不被默认调度。

```
启动期主导：建立骨架       │ 偶尔回到此期：问题树重构、生成新透镜
探索期主导：填充和深挖     │ 是系统大部分时间所在的期
收敛期主导：收拢和输出     │ 偶尔回到探索期：补关键证据
```

这个设计的核心意图是：保持流程的可预期性，同时不牺牲分析的灵活性。用户和 Orchestrator 都能清楚知道"系统现在主要在做什么"，但系统不会因为流程的形式约束而错过真正重要的回修机会。

## 前端设计

### 设计哲学

前端最大的诚实是节制：默认隐藏过程，按需暴露细节；先抛洞察，再讲证据；标明可靠性边界，让用户自己决定信多少。后端越复杂，前端越要简洁——这是对后端工作的最大尊重。

核心反原则：不要把后端有的都展示出来。系统的核心价值是产出张力式洞察——一个让用户"啊，原来如此"的瞬间。如果前端把所有过程信息都铺给用户，用户会被淹没在过程里，错过结论本身。真实分析员交付给客户的不是"我做了哪些事"，而是"我得出了什么、为什么、可信度多少"。

### 前端要回答的三个根本问题

1. "系统现在在做什么？" —— 过程中的可观察性，避免黑箱焦虑
2. "系统给了我什么？" —— 结果的可消化性，让洞察真的传达到
3. "我能信多少、能改什么？" —— 可信度和可介入性，让用户保持主导权

下面分别展开。

------

### 过程阶段：焦虑管理，而非信息完整呈现

分析过程可能跑几分钟到几十分钟，用户不会盯着看。前端在过程阶段要解决的是焦虑管理。

默认可见的内容：

- 当前所在的期：启动期 / 探索期 / 收敛期，配进度条或时间线
- Orchestrator 实时旁白：一句话描述系统正在做什么（如"正在对子问题 2 做横向类比搜索"）。由 Scheduler 在每次调度时写出
- 问题树状态：每个子问题的状态用颜色或图标表示（untouched / exploring / closed / stuck）
- 关键里程碑事件流：问题树建立、初始透镜生成、第一条 structure 层发现、透镜演化、子问题闭合等——这些是用户能感知到"系统在推进"的标志事件

按需展开的内容（折叠在二级面板）：

- 黑板各分区的实时内容
- 透镜的演化轨迹
- Agent 之间的辩论记录
- 调度日志

绝对不默认展示的内容：

- 原始 LLM 调用日志、token 消耗、API 错误
- 每条证据的全部元数据
- Orchestrator 的内部状态

类比：过程阶段的前端应该像直播间的弹幕区——主屏幕是核心进展的简洁呈现，弹幕是次要信息流，用户想看就看。

过程阶段的布局示意：

```
┌────────────────────────────────────────┐
│  当前期 + 时间线（启动→探索→收敛）          │
│  [████████░░░░░░░] 探索期 60%           │
├────────────────────────────────────────┤
│  Orchestrator 旁白：                    │
│  "正在对子问题 2 做横向类比搜索"           │
├────────────────────────────────────────┤
│  问题树状态：                            │
│  driving_question                      │
│  ├── 子问题 1 ✓ closed                   │
│  ├── 子问题 2 ⟳ exploring                │
│  └── 子问题 3 ⊘ stuck                    │
├────────────────────────────────────────┤
│  里程碑事件流（最新在上）：                │
│  · +14:32 第一次透镜演化（v1 → v2）        │
│  · +07:28 子问题 1 闭合                   │
│  · +00:00 启动期完成                      │
├────────────────────────────────────────┤
│  [打断 / 调整偏好] 按钮                   │
└────────────────────────────────────────┘

```

------

### 结果阶段：按用户的认知节奏组织信息

这是前端最关键的部分。系统输出包含五项：convergent_finding、tension、boundary_condition、unresolved、implication。

展示顺序不应该按字段定义顺序，而应该按用户的认知节奏：

1. 先抛核心结论：对 driving_question 的直接回答（一两句话）
2. 然后是张力（tension）：洞察密度最高的部分，最容易触发"啊哈"瞬间
3. 再是边界条件（boundary_condition）：分析员的诚实——结论在哪里成立、哪里失效
4. 接着是支撑证据（convergent_finding）：多角度的收敛结论
5. 最后是未尽之处（unresolved）：明确"什么还没回答"

为什么这个顺序：

用户的注意力是稀缺的，最重要的洞察必须放在前面。convergent_finding 虽然是"证据最扎实的部分"，但信息密度低、惊喜度低，放在前面会让用户失去阅读耐心。

tension 放在第二位是有讲究的：用户看到结论后会本能地想"凭什么这么说？"——这时立即抛 tension（"这个结论有这些矛盾，但我们的判断是这样平衡的"）比立即抛证据更能建立信任。这是先暴露弱点再展示力量的反直觉技巧，专业咨询报告通常就是这么组织的。

渐进披露原则：

- 默认折叠状态：只看 driving_question 的最终回答 + 各子问题的标题和一句话核心结论
- 点击展开：看到该子问题的完整五项
- 每条结论旁边有"溯源"按钮：点击可以看到这条结论基于哪些证据、来自哪个透镜版本

这种渐进披露让不同深度的读者都能找到自己想要的信息——5 分钟读完核心，15 分钟读完完整分析。

结果阶段的布局示意：

```
┌────────────────────────────────────────┐
│  顶部：driving_question + 核心回答       │
│  （永远可见，结果出来就在这）              │
├────────────────────────────────────────┤
│  分析诚信度摘要（卡片式，简洁）             │
├────────────────────────────────────────┤
│  子问题区（默认折叠，可逐个展开）           │
│  ├── 子问题 1：[核心结论一句话]           │
│  │   [点击展开看完整 5 项]               │
│  ├── 子问题 2：...                      │
│  └── ...                               │
├────────────────────────────────────────┤
│  最终洞察（综合 implication）             │
│  边界条件 + 未尽之处                      │
├────────────────────────────────────────┤
│  侧栏（可隐藏）：                         │
│  - 透镜演化轨迹                          │
│  - 关键辩论摘要                          │
│  - 调度日志（默认折叠）                   │
└────────────────────────────────────────┘

```

------

### 可信度的显性化

这是本系统区别于普通 AI 工具的关键。普通 AI 工具的输出是一个黑箱"答案"，本系统输出的是带有诚实标注的分析。前端必须把这种诚实显性化。

分析诚信度摘要（在结果顶部展示）：

- 子问题闭合情况：X 个 closed，Y 个 stuck
- 关键 killer_evidence 找到情况：找到 N 条，未找到 M 条
- LLM 降级情况：本次分析中有 X 处决策走了降级路径
- 透镜稳定性：最终透镜经历了几次大演化

这个摘要的价值是让用户知道结论的可靠性边界——比"输出一段听起来很自信的话"诚实得多。

------

### 可介入性：设计成断点确认 + 异步通知

"用户随时调整"在异步任务里是个虚假承诺——用户不会全程盯着分析过程。前端不应承诺"实时控制"，而应设计成断点确认 + 异步通知：

- 在启动期结束、探索期中期、收敛期开始前各设一个软断点
- 默认自动通过；用户可在设置里开启"在断点暂停等待我"
- 系统在断点处发送通知（浏览器推送 / 邮件 / 站内消息），用户决定是否介入

用户介入点的具体形式：

- 启动期确认：展示初始问题树、透镜、预判，允许用户修改、追加、删除
- 过程中调整：允许用户改变远近偏好、追加追问方向
- 结果阶段补刀：用户对某个 tension 不满足时，可以指令系统针对性地补刀

这样既保留了用户主导权，又不要求用户全程在场。

------

### 关于"过程可视化"的克制原则

把问题树、透镜演化、黑板分区做成炫酷的可视化（动态图、版本链、关系图）是一种诱惑，但是项目的最大陷阱之一。

理由：

- 这些可视化做好需要大量前端工程，但对核心价值贡献低——用户要的是洞察，不是过程的视觉呈现
- 过程可视化容易让人误以为"看起来很科学，所以结论很可靠"——这是一种诱导性的可视化暴力
- 真正想看过程的是开发者（调试用）和少数高阶用户——不是大多数用户

实施原则：

- MVP 阶段，过程可视化只做最简版（文字日志 + 问题树状态颜色），不投入前端工程力气
- 高级可视化放到"开发者视图"或"高阶用户视图"里，默认不开启
- 真正花心思的地方是结果展示——清晰的层级、合理的折叠、舒服的阅读节奏

------

### 一句话总结

前端的核心设计原则：默认隐藏过程，按需暴露细节；先抛洞察，再讲证据；标明可靠性边界，让用户自己决定信多少。

最大的反例：把所有中间状态做成炫酷 dashboard，让用户感觉"哇好复杂好牛"，但 10 分钟过去他还没看明白结论是什么。这是把分析工具做成了演示玩具。

## 技术栈

- LLM: DeepSeek (OpenAI-compatible API)
- Agent 框架: LangGraph
- 搜索: Serper API
- Web: Flask + SSE
- 前端: 原生 HTML/CSS/JS
- 数据模型: Pydantic v2

> 选择 LangGraph 而非 CrewAI 的原因：CrewAI 的最小调度单位是整个 Agent 回合，无法在 Agent 之间插入精细的调度决策点（即 CrewAI 一次任务调用就是一个完整 Agent 执行流程，Orchestrator 无法在中途介入）；LangGraph 的 StateGraph 允许以节点为粒度，每个任务完成后 Orchestrator 立即重评估，天然匹配本系统的短任务 + 高频调度循环。

## 开发命令

```bash
# 安装依赖
python3 -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt

# CLI 运行
python main.py "AI 公司现在应不应该烧钱扩张？"  # 基础用法（只传问题）

# 完整用法（问题 + 情境 + 偏好）
python main.py \
  --question "AI 公司现在应不应该烧钱扩张？" \
  --context "我在评估投资某 AI 初创" \
  --mode focus  # focus | balance | explore


# Web 运行
python app.py  # http://localhost:5001
```
