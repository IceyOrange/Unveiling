# PRD.md

## 项目简介

Unveiling 是一个多 Agent 类比分析系统。用户带着问题或情境来（而非孤立主题），系统通过横向（跨领域）和纵向（跨时期）类比，以及抽象化的透镜思维，揭示事物深层结构性模式，产出张力式洞察。

## 核心哲学

### 根基

**1. 结构比表面更真实（解释力来自结构层，不来自现象层）**

"深层结构上的映射"比"表面相似"更有价值。如: AI 和印刷术都"引发了公众恐慌"是现象层共性；"都使某类认知劳动的边际成本趋近于零"是结构层共性——后者有迁移性和预测力，前者没有。

> 为什么结构层有迁移性——因为结构层描述的是关系和机制，而关系和机制可以在不同的表面材质上实例化。

**2. 事物的普遍性**

万物在深层结构上存在跨时空的映射。面对任何问题，时空的另一角落往往存在结构相似且值得寻找的案例。通过研究这些远距离案例，识别其深层结构，映射回当前问题，可以洞见当前问题的本质。但如果当找到的类比都停留在现象层、无法提供结构性理解时，这本身就是一个输出——说明这个问题可能是真正的新事物。

**3. 距离产生清晰（"不识庐山真面目，只缘身在此山中"）**

为什么不直接分析当前问题？因为身处其中时，偏见和盲区阻碍了真正的理解。借助时空中的远距离案例进行类比，我们获得外部视角，才能看到身处其中时看不到的结构。

"类比是认识的**入口**，直接分析是类比之后的**落地**。系统优先通过类比建立结构性理解，在此基础上才触碰当前问题的具体细节——而不是反过来。"

**4. 远近张力原则**

认知价值和认知可达性之间存在内在张力：最有价值的类比往往最难被识别为相关。透镜是这个张力的操作解：它不消除距离，而是在保持距离的同时建立结构映射，让遥远的案例在抽象层上变得可比。

透镜本身随分析过程演化——初始透镜是进入的入口，演化后的透镜是分析的结论之一。其演化由两种力量驱动：近搜索带来的结构确认，和远搜索带来的结构挑战。当一个远距离案例无法被当前透镜解释但直觉上重要时，这是透镜需要演化的信号。

基于该原则，系统应该显式地同时维护"远"和"近"两种搜索模式，而不是只有一种。

- "近"搜索：在当前透镜框架内寻找结构匹配的类比——高命中率，低惊喜
- "远"搜索：主动寻找当前透镜框架解释不了、但直觉上感觉有趣的案例——低命中率，高潜在价值

两种搜索的配比，就是"问题导向"和"自由联想"张力的实操版 trade-off。可以把它变成一个可调的旋钮，就像是模型的 temperature 参数一样。

### 衍生原则

基于上述根基：

**1. 工具论：用什么思考**

时空类比和透镜是系统的核心认知工具，不绑定到特定阶段，随时可调用。

两者的关系是：类比提供原材料，透镜提供结构映射的框架。没有透镜，遥远的类比无法被评估；没有类比，透镜是空转的抽象。

透镜不是分析的前提，而是分析的产物。初始透镜是进入问题的脚手架，随着类比证据的积累，透镜被修正、分裂或合并——其演化轨迹本身记录了理解的发生过程，是洞察的一部分。

**2. 证据论：如何对待发现**

所有发现的共性都不丢弃，但追问方式因性质而异。

系统不预先判断一条共性的真假，而是追问三个问题：它在哪个层级上成立（phenomenon、mechanism、还是 structure）？在当前问题框架下是否有解释力？它为什么看起来成立——或为什么看起来不成立？

这个追问过程本身产生洞察，无论最终结论如何：真共性揭示结构，假共性揭示认知盲区，层级判断本身校准透镜。

**3. 导向论：以什么为罗盘**

以问题为锚，以结构性理解为目标，以意外发现为信号。

"以问题为锚"不意味着只追求直接相关的类比。"看起来无关"本身是一个需要透镜才能做出的判断，而透镜还在演化中——过早放弃远距离类比，可能是用一个尚未成熟的透镜关上了一扇门。

因此：近搜索（透镜框架内的结构匹配）保证问题被回答，远搜索（框架外的意外发现）保证透镜被挑战。两者的配比是可调的，但两者都不可缺席。

## 架构方向

### 设计原点

核心哲学确立了两件事：分析应以问题为锚，以结构性理解为目标。系统分为三个阶段：抽象、搜集、收拢。抽象是贯穿全流程的认知工具——阶段一用它翻译问题，阶段二可能用它理解意外发现，阶段三用它归纳共性。

### 三阶段生命周期

一次完整分析分三个阶段，每个阶段职责清晰、不重叠：

```
阶段一：抽象                    阶段二：搜集                阶段三：收拢
──────────────               ──────────────              ──────────────
输入：用户问题                  输入：抽象概念               输入：~20 个案例
识别主体/关系                   横向搜索 ←→ 纵向搜索        跨案例归纳共性
抽象到结构层                    （并行，黑板协作）           发现张力
                              各找 10 个或超时             质疑检验
输出：结构性抽象概念             输出：证据区积累的案例       组织结论
```

三阶段是单向流水线，不回流。每个阶段只做自己的事：抽象不试图发现深层机制（信息不足），搜集不试图收拢结论（证据不全），收拢不回头重新搜索（预算有限）。

------

### 阶段一：抽象

**目标**：把用户的具体问题翻译成能跨时空搜索的结构性语言。

**入口条件**：用户输入问题。

**过程**：

1. 识别问题中的主体（entity）和主体间的关系（relationship）
2. 将每个主体抽象到结构层——到可跨时空比较的粒度为止
3. 将关系抽象到结构层

**示例**：

```
输入："AI 时代人们的 AI 焦虑"

第一步：识别主体和关系
  主体：AI、人们、焦虑
  关系：AI → 让 → 人们 → 产生焦虑

第二步：逐个抽象到结构层
  AI   → [LLM 判断的结构角色，如"崛起中的、能力可外溢的新型生产力"]
  人们 → [LLM 判断的结构角色，如"其价值与身份正被新生产力冲击的主流主体"]
  焦虑 → [LLM 判断的结构角色，如"主体面对外部不可控变化时的存在论反应"]
  关系 → [LLM 判断的结构关系，如"新生产力 → 渗透/重构主体的功能与意义领地 → 触发主体的存在论焦虑"]
```

**关键约束**：问题中每一个出现的主体和关系都必须进入抽象结果，不能因为"看起来只是修饰词"就略去。`焦虑` 是问题本身的焦点现象，必须作为独立主体被抽象，而不是隐入关系字段——否则阶段二只会搜到"替代关系"的类比，丢失"群体性焦虑"这条搜索线。

抽象到什么程度、抽象成什么，由 LLM 根据具体问题判断，不预设映射。

**不做的事**：

- 不追问更深层结构（证据还没收集，深层归纳没有基础）
- 不拆子问题（探索方向从抽象结果自然长出）
- 不下预判（过早）
- 不做两步抽象（只做一步，到结构层就停）

**输出**：一个结构层的抽象概念（= 初始透镜），包含：
- 各主体的结构角色
- 主体间关系的结构模式

**出口条件**：抽象概念生成完成，写入黑板假设区。

------

### 阶段二：搜集

**目标**：拿着阶段一产出的抽象概念，去时空里广泛搜集结构匹配的实例。

**入口条件**：阶段一完成，抽象概念写入黑板。

**过程**：

横向搜索（当代跨领域）和纵向搜索（跨时期）**并行运行**，通过黑板实时协作：

**方向定义**：

| 方向 | 跨什么 | 不跨什么 | 例子 |
|------|--------|----------|------|
| 横向 | 跨领域（不同行业/学科） | 不跨时间，必须是**当代/当下**现象 | 石油行业抵制电动车、社交媒体焦虑、基因编辑争议 |
| 纵向 | 跨时间（不同历史时期） | —（可以跨领域） | 卢德运动 1810s、印刷术革命 1450s、电脑普及 1980s |

**搜索流程**（每方向独立循环）：

```
抽象概念（阶段一产出）
        │
    ┌───┴───┐
    ▼       ▼
  横向搜索  纵向搜索      ← 并行，各自独立收敛
    │       │
    ▼       ▼
  生成查询词                ← 读黑板已有案例，避免重复
    │
    ▼
  LLM 验证查询词            ← 搜索前预判：会重复吗？方向对吗？
    │
    ▼
  调 Serper 搜索
    │
    ▼
  提取案例                  ← 1 个案例 = 1 条证据记录（带 case_name）
    │
    ▼
  写黑板                    ← 提交证据，对方下一轮可读到
    │
    ▼
  满 10 个或跑满 3 轮？→ 该方向收敛
```

**证据粒度**：一个独立案例（如"卢德运动"）= 一条 EvidenceRecord，带 `case_name` 字段。不把同一案例拆成多条记录。

**黑板反馈机制**：
- 每轮搜索开始前读取黑板全部案例（含对方方向的发现）
- 查询词生成利用已有案例作为 insight，避免重复、发现新方向
- 搜索前 LLM 验证层拦截重复和方向错误（如横向搜到历史案例）

**不做的事**：

- 不在搜集阶段修改透镜（一边找一边换罗盘反而搜集不到广度）
- 不质疑评判（质疑和评判放到阶段三，拿着全部证据一起审更有意义）
- 不深挖特定方向（广度优先，深度是阶段三的事）

**退出条件**（每方向独立收敛）：

- 某方向找到 10 个案例 → 该方向停止搜索
- 某方向跑满 3 轮仍未到 10 → 该方向带着现有结果收敛
- 两个方向都收敛 → 进入阶段三
- 横向和纵向互不阻塞：一方先收敛，另一方继续搜索

**输出**：证据区积累了跨时空的结构匹配案例。

------

### 阶段三：收拢

**目标**：根据阶段二收集到的信息，归纳深层结构，产出张力式结论。

**入口条件**：阶段二退出条件达成。

**过程**（串行管线，每步输出是下一步输入）：

**第一步：归纳共性**

跨案例抽象——不是逐个抽象每个案例，而是从全部证据中找出结构性共同点。底层机制是什么？

这是阶段一故意没做的「追问更深层」。现在有证据了，可以做。

**第二步：发现张力**

从归纳出的深层结构里找真正的结构性冲突——不是「有正反两面」，而是两股力量在结构上互斥又共存。

**第三步：质疑检验**

回头看全部案例：有没有不匹配的？归纳出的共性有没有确认偏误？带着全局视角质疑，比在搜集阶段零散质疑更有力。

**第四步：组织结论**

最终输出，包含：
- **核心发现**：最稳健的结构性洞察
- **张力**：真正的结构性冲突
- **边界条件**：结论在什么条件下成立/失效
- **未解决的**：诚实标注的不确定项
- **启示**：对原始问题的回答

**退出条件**：结论写入黑板结论区。

------

### 抽象：贯穿全流程的认知工具

抽象不只在阶段一使用，而是系统的核心认知原语：

- **阶段一**：抽象用户问题 → 得到搜索罗盘
- **阶段二**（例外情况）：搜索到意外案例时，抽象它能理解其结构角色
- **阶段三**：抽象全部证据 → 归纳深层共性

------

### 全局黑板

黑板是共享状态，分四个分区：

- **假设区**：透镜（初始抽象概念 + 演化版本）。透镜记录主体/关系的结构角色。版本链完整保留。
- **证据区**：搜索结果、具体案例。每条带元数据：`case_name`（案例名称）、来源透镜、搜索方向（横向/纵向）、层级标签（phenomenon / mechanism / structure）、置信度（strong / medium / weak / unexpected）。一个案例 = 一条记录。只增不删。
- **结论区**：阶段三写入的张力式输出。
- **调度日志区**：关键决策痕迹（阶段切换、LLM 失败降级）。只增不删。

#### 黑板的并行协作机制

阶段二的横向和纵向搜索通过黑板协作：

- 各自独立搜索，分批提交结果到证据区
- 每轮搜索开始前读取黑板最新状态，可参考对方发现
- 延迟在 LLM 调用粒度内，感知上接近实时

#### 记录状态

每条黑板记录的状态：

- **committed**：记录已稳定，可安全引用。进入黑板即默认 committed。
- **retracted**：已撤回，记录保留（追溯性），标记撤回原因。

------

### Orchestrator

Orchestrator 是有限状态机 + LLM 判断点的混合控制器，负责阶段间的流转和搜索调度。

#### 阶段一 Orchestrator

极简——阶段一完成后直接进入阶段二。无需调度决策。

#### 阶段二 Orchestrator（Scheduler）

调度逻辑：

- 跟踪搜索覆盖率：横向找到几个、纵向找到几个、各跑了几轮
- 每方向独立收敛：到 10 个或跑满 3 轮即停
- 当两个方向都收敛时，进入阶段三
- 搜索前有 LLM 验证层：拦截重复查询和方向错误

#### 阶段三 Orchestrator

无需调度——阶段三是串行管线，按固定步骤执行。

#### 失败降级

通用原则：LLM 失败时降级为保守默认值，继续运行而不终止，在调度日志区标记降级事件。

| 失败场景 | 降级行为 |
|:---|:---|
| 阶段一抽象失败 | 用用户原始问题作为透镜，直接进入阶段二 |
| 搜索 agent LLM 失败 | 跳过本轮搜索，记录降级事件 |
| 阶段三收敛失败 | 生成最小结论（标注"收敛失败"） |

降级标记在最终输出中呈现给用户，让用户知道结果的可靠性边界。

#### 与 LangGraph 的对接

- 阶段一：单节点（inception_node）
- 阶段二：并行分支（parallel_search_node 内部运行 lateral + vertical），循环直到双方收敛
- 阶段三：单节点（convergence_node）
- 条件路由：基于各方向独立收敛状态决定何时退出阶段二

### 收敛与输出

输出结构为张力式，不是共性堆叠：

- **核心发现**（core_finding）：最稳健的结构性洞察
- **张力**（tension）：真正的结构性冲突。找不到张力 → 显式输出"分析未发现结构性张力"，不用共性堆叠掩盖
- **边界条件**（boundary_condition）：结论在什么条件下成立/失效，具体到可测试
- **未解决的**（unresolved）：强制项。没有 unresolved → 标注"分析可能过度自信"
- **启示**（implication）：对原始问题的回答

## 前端设计

### 设计哲学

前端最大的诚实是节制：默认隐藏过程，按需暴露细节；先抛洞察，再讲证据；标明可靠性边界，让用户自己决定信多少。后端越复杂，前端越要简洁——这是对后端工作的最大尊重。

核心反原则：不要把后端有的都展示出来。系统的核心价值是产出张力式洞察——一个让用户"啊，原来如此"的瞬间。如果前端把所有过程信息都铺给用户，用户会被淹没在过程里，错过结论本身。

### 前端要回答的三个根本问题

1. "系统现在在做什么？" —— 过程中的可观察性，避免黑箱焦虑
2. "系统给了我什么？" —— 结果的可消化性，让洞察真的传达到
3. "我能信多少？" —— 可信度的显性化

------

### 过程阶段

分析过程可能跑几分钟到几十分钟。前端在过程阶段解决焦虑管理。

默认可见的内容：

- 当前阶段：抽象 → 搜集 → 收拢，配进度条
- 实时旁白：一句话描述系统正在做什么
- 搜索进度：横向 X/10，纵向 Y/10
- 关键里程碑事件流

按需展开的内容（折叠在二级面板）：

- 证据区的实时内容
- 透镜信息
- 调度日志

------

### 结果阶段

展示顺序按用户的认知节奏：

1. **核心结论**：对原始问题的直接回答（一两句话）
2. **张力**：洞察密度最高的部分
3. **边界条件**：结论在哪里成立、哪里失效
4. **核心发现**：支撑证据
5. **未解决之处**：什么还没回答

渐进披露：默认只看核心结论 + 张力，点击展开看完整五项。

------

### 分析诚信度摘要

在结果顶部展示：

- 搜索覆盖情况：横向 N 个、纵向 M 个案例
- LLM 降级情况：本次分析中有 X 处决策走了降级路径
- 超时情况：是否因超时提前退出搜集

### 关于"过程可视化"的克制原则

MVP 阶段，过程可视化只做最简版（文字日志 + 搜索进度），不投入前端工程力气。真正花心思的地方是结果展示——清晰的层级、合理的折叠、舒服的阅读节奏。

## 技术栈

- LLM: DeepSeek (OpenAI-compatible API)
- Agent 框架: LangGraph
- 搜索: Serper API
- Web: Flask + SSE
- 前端: 原生 HTML/CSS/JS
- 数据模型: Pydantic v2

> 选择 LangGraph 而非 CrewAI 的原因：CrewAI 的最小调度单位是整个 Agent 回合，无法在 Agent 之间插入精细的调度决策点；LangGraph 的 StateGraph 允许以节点为粒度，每个任务完成后 Orchestrator 立即重评估，天然匹配本系统的短任务 + 高频调度循环。

## 开发命令

```bash
# 安装依赖
python3 -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt

# CLI 运行
python main.py "AI 时代人们的 AI 焦虑"

# 完整用法（问题 + 情境 + 偏好）
python main.py \
  --question "AI 时代人们的 AI 焦虑" \
  --context "我在写一篇关于技术焦虑的专栏" \
  --mode focus  # focus | balance | explore

# Web 运行
python app.py  # http://localhost:5001
```

## 环境变量

配置在 `.env` 中：
- `OPENAI_API_KEY` — DeepSeek API key
- `OPENAI_API_BASE` — https://api.deepseek.com
- `OPENAI_MODEL_NAME` — deepseek-v4-flash（deepseek-v4-pro 可选，质量更高）
- `SERPER_API_KEY` — Serper API key
